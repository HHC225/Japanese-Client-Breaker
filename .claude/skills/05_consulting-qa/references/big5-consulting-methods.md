## McKinsey — Structure & Logic {#mckinsey}

### Core Framework: The Pyramid Principle (Barbara Minto)

McKinsey's communication standard, developed by Barbara Minto, demands that every argument follow a top-down pyramid structure:

```
         [Governing Thought]
              /    |    \
       [Key Point 1] [Key Point 2] [Key Point 3]
       /    |    \
  [Support] [Support] [Support]
```

**Rules**:
1. **Start with the answer** — The governing thought comes first, not last
2. **Group and summarize** — Supporting points are grouped logically
3. **MECE at every level** — Each group of supports must be mutually exclusive and collectively exhaustive
4. **"So What?" at every level** — Every statement must answer "Why should the audience care?"

### Evaluation Criteria (Detailed)

#### Pyramid Compliance
- Does the argument start with the conclusion?
- Are supporting points organized under clear group headings?
- Is the hierarchy clean (no more than 3-4 levels)?

**Grade A**: Arguments read top-down naturally. Reader grasps the core message in the first sentence.
**Grade F**: Arguments build up to a conclusion buried at the end. Reader must work to find the point.

#### MECE Validation
- Are supporting points truly non-overlapping?
- Do they cover the full space of the argument?

**Grade A**: Supporting points are clearly distinct and complete. No gaps or overlaps.
**Grade F**: Points overlap, or obvious categories are missing.

#### "So What?" Test
Apply at each level of the argument:
- Level 1 (your argument): "So what?" → should lead to "Therefore, the client should accept because..."
- Level 2 (your counter): "So what?" → should lead to "Therefore, the initial concern is addressed because..."
- Level 3 (your resolution): "So what?" → should lead to "Therefore, we can move forward with..."

**Grade A**: Every statement has a clear, non-trivial implication for the audience.
**Grade F**: Statements are descriptive without connecting to audience impact.

#### Issue Tree Integrity
Can the argument be drawn as a clean issue tree?
- No circular dependencies
- No orphan branches
- Every branch has a resolution

### McKinsey Communication Standards
- Use **SCQA (Situation-Complication-Question-Answer)** framing for introductions
- **Horizontal Logic**: Reading titles alone must form a complete persuasive argument
- **Vertical Logic**: Each section's content must directly prove its title assertion
- **Action Titles**: Active sentences stating conclusions (max 15 words), not topic labels
- Quantify every claim: "improved" → "improved by 23% over 6 months"
- Limit each level to 3-5 supporting points (cognitive load management)
- **Obligation to Dissent**: Every argument must survive rigorous internal challenge; complacent acceptance is a quality failure
- **Straw Man Protocol**: Present arguments as deliberately imperfect drafts, collect all critiques, iterate to final version

### What a McKinsey Reviewer Would Say

**Passing argument**: "Clear, well-structured, data-driven. The pyramid is clean and every level passes the 'So What?' test."

**Failing argument**: "I can't find the punchline. The argument meanders through details without a clear governing thought. The supporting points overlap and there's no MECE structure."

## BCG — Strategic Pragmatism {#bcg}

### Core Framework: Scenario-Based Strategic Thinking

BCG's hallmark is combining rigorous analysis with strategic pragmatism. Arguments must work in the real world, not just on paper.

### Evaluation Criteria (Detailed)

#### Scenario Robustness
Test each argument against 3 scenarios:
1. **Best case**: Does the argument still make sense if everything goes right? (No over-engineering for risks that don't materialize)
2. **Worst case**: Does the argument hold if things go wrong? (Not naive optimism)
3. **Most likely case**: Is this practical for the expected reality?

**Grade A**: Argument explicitly addresses multiple scenarios and remains valid across all.
**Grade F**: Argument only works under ideal conditions.

#### Pragmatic Feasibility
- Is the proposed defense actually executable in a real meeting?
- Can the evidence be gathered in time?
- Is the nemawashi strategy realistic given organizational dynamics?

**Grade A**: Could execute this defense tomorrow in a real meeting with no additional preparation.
**Grade F**: Relies on evidence or stakeholder alignment that doesn't exist yet.

#### Trade-off Awareness
BCG values honesty about trade-offs over pretending everything is optimal.

**Grade A**: "We chose X over Y because of [trade-off]. Y has advantages in [areas] but X optimizes for [client's priority]."
**Grade F**: "X is the best option with no downsides." (BCG would call this "lazy thinking")

#### Novel Insight Requirement (BCG Differentiator)
BCG demands that analysis generates a NEW insight the client doesn't already have. Restating the obvious is not value-add.

- Does the argument tell the client something they didn't know before?
- Does it reframe the problem in a way that opens new possibilities?
- Would a BCG partner say "that's interesting" or "I already knew that"?

**Grade A**: Analysis surfaces a genuinely new perspective or reframes the problem — the client's thinking shifts.
**Grade F**: Restates what the client already knows in consultant language. No new insight generated.

### BCG Additional Frameworks
- **Strategy Palette**: 5 approaches (Classical, Adaptive, Visionary, Shaping, Renewal) — argument must match the client's strategic environment
- **SCQA Narrative Arc**: Situation → Complication → Question → Answer — creates urgency before the recommendation
- **Novel Insight Requirement**: Analysis must generate a NEW insight, not just confirm what the client already knows

### What a BCG Reviewer Would Say

**Passing argument**: "Strategically sound, honest about trade-offs, and practically executable. Good scenario testing."

**Failing argument**: "This is theoretically elegant but ignores practical constraints. No scenario testing means it only works if everything goes perfectly."

## Bain — Results Orientation {#bain}

### Core Framework: Results-First, Action-Oriented

Bain's philosophy centers on delivering measurable results. Every argument must connect to a concrete outcome. Bain invented the **Results Delivery** methodology — positive reinforcement is **4x more powerful** than push activities in changing behavior.

### Evaluation Criteria (Detailed)

#### Value Bridge (Bain Results Delivery® Core)
Every argument must connect to a quantified target decomposed into drivers and initiatives.

**Grade A**: "Upon acceptance, [quantified target] is achieved by [date] via [decomposed drivers]. Finance-verifiable."
**Grade F**: The argument ends with acceptance but no quantified outcome or decomposition.

Source: [Bain Results Delivery®](https://www.bain.com/consulting-services/change-management-results-delivery/results-delivery-office/)

#### Adoption Risk Diagnostics (Bain RD® Differentiator)
Bain starts by profiling where behavior change is hardest — by function, site, role. Risks are made visible upfront.

- Are adoption risks identified? Who must change? How hard will it be?
- Is organizational readiness assessed?

**Grade A**: "Adoption risks identified: [role X] faces [specific change], difficulty: [HIGH]. Mitigation: [plan]."
**Grade F**: No consideration of adoption barriers. Assumes frictionless implementation.

Source: [Bain Adoption Risk Diagnostics](https://umbrex.com/resources/frameworks/project-management-frameworks/bain-results-delivery/)

#### Actionability
- Can the client act on this immediately?
- Are responsibilities clear with who/what/when?

**Grade A**: "Next steps: [Person A] does [X] by [date]. [Person B] does [Y] by [date]. Review at [date]."
**Grade F**: "Further consideration is recommended." (Bain would reject this as non-actionable)

#### Outcome Measurability
Leading and lagging indicators must be defined. Benefits must be Finance-verifiable.

**Grade A**: "Leading indicator: [metric] tracked weekly. Lagging indicator: [KPI] at 90-day review. Finance sign-off required."
**Grade F**: Arguments presented without measurable success criteria.

### Bain Additional Frameworks
- **Adoption Risk Diagnostics**: Before recommending, identify and quantify WHERE adoption will falter — making resistance visible upfront
- **Influencer Activation**: Map key influencers in client organization and design specific activation strategies for each
- **Value Bridge**: Clear traceability from recommendation → actions → realized value (not just theoretical benefit)
- **NPS Lens**: Does the recommendation make the client an advocate? Long-term relationship health matters.

### What a Bain Reviewer Would Say

**Passing argument**: "Clear outcomes, actionable next steps, measurable success criteria. I know exactly what happens if the client accepts."

**Failing argument**: "Interesting analysis but it doesn't go anywhere. What's the action? Who does what? When?"

## Deloitte — Evidence & Compliance {#deloitte}

### Core Framework: Audit-Grade Evidence Standards

Deloitte's Big 4 heritage means everything must be auditable and compliant. In a Japanese banking context, this is especially critical.

### Evaluation Criteria (Detailed)

#### Evidence Quality Assessment

| Level | Description | Example |
|-------|-------------|---------|
| **Primary** | Original data you generated/collected | Survey results, test data, measurements |
| **Secondary** | Published data from credible sources | Industry reports, regulatory docs, academic papers |
| **Tertiary** | Expert opinions or general knowledge | "Best practices suggest..." |

**Grade A**: Arguments primarily use Primary and Secondary evidence. Every claim has a specific citation.
**Grade F**: Arguments rely on tertiary evidence or make claims without any source.

#### Regulatory Awareness
For Japanese banking context, arguments must consider:
- 金融庁 (FSA) guidelines
- FISC安全対策基準 (FISC Security Standards)
- バーゼルIII (Basel III) requirements
- 個人情報保護法 (Personal Information Protection Act)
- J-SOX compliance
- 反社会的勢力排除 (Anti-social forces exclusion) requirements

**Grade A**: Regulatory implications explicitly addressed with specific article/section references.
**Grade F**: No mention of regulatory context in a banking deliverable.

#### Audit Trail
- Can each claim be traced back to its source document?
- Are dates, versions, and page numbers cited?
- Is the evidence chain unbroken?

**Grade A**: Full citation trail: "[Claim] (Source: [Document], [Version], [Section], [Date])"
**Grade F**: "[Claim]" with no source attribution.

#### Risk Documentation
Deloitte demands formal risk assessment for every significant argument:

| Risk Aspect | Required Content |
|-------------|-----------------|
| Risk ID | Unique identifier |
| Description | Clear statement of the risk |
| Probability | Quantified likelihood |
| Impact | Quantified consequence |
| Mitigation | Specific countermeasure |
| Owner | Who is responsible |
| Status | Current state |

#### Industry Benchmarking
- Are claims compared against industry peers?
- Are benchmarks from credible, recent sources?
- Is the comparison methodology transparent?

### Deloitte Additional Frameworks
- **Enterprise Value Delivery (EVD)**: Built on ITIL/ITSM/CMMI — integrates PM, QM, change management, testing. QA advisor conducts quarterly reviews.
- **Strategic Choice Cascade** (Roger Martin): 5 questions every recommendation must answer: (1) Winning aspiration? (2) Where to play? (3) How to win? (4) Required capabilities? (5) Management systems? — WTP and HTW must be a matched pair.
- **Definitions of Ready/Done**: Every deliverable must have explicit, testable acceptance criteria defined upfront.

### What a Deloitte Reviewer Would Say

**Passing argument**: "Every claim is evidenced, regulatory implications are considered, and the risk framework is sound."

**Failing argument**: "Where are the sources? How would this hold up in a regulatory examination? The risk assessment is incomplete."

## Accenture — Implementation & Technology {#accenture}

### Core Framework: End-to-End Implementation Readiness

Accenture focuses on the gap between strategy and execution. Their **Accenture Delivery Methods (ADM)** framework spans strategy through execution, blending agile, DevOps, and classic PM with common governance, quality, and risk controls. Arguments must be implementable.

### Evaluation Criteria (Detailed)

#### Implementation Feasibility
- Is the proposed approach technically feasible?
- Are there known blockers or dependencies?
- Has a proof of concept been considered?

**Grade A**: Clear implementation plan with identified dependencies, blockers addressed, and contingencies defined.
**Grade F**: Theoretical approach with no implementation consideration.

#### Technology Alignment
- Does the approach align with the client's existing technology stack?
- Are integration points identified?
- Are technology risks addressed?

**Grade A**: "This approach integrates with the client's existing [system] via [method], requiring [specific changes]."
**Grade F**: Technology implications not considered.

#### Resource Realism
- Are effort estimates realistic?
- Are skill requirements identified?
- Is the resource availability confirmed?

**Grade A**: "[X] person-hours estimated, requiring [skills]. Team allocation confirmed for [timeline]."
**Grade F**: No resource estimation or wildly optimistic estimates.

#### Change Management
Accenture emphasizes the human side of implementation:
- Who is impacted by this change?
- What training is needed?
- How will adoption be measured?
- What resistance is expected?

**Grade A**: Change impact assessment with stakeholder map, training plan, and adoption metrics.
**Grade F**: Technical solution presented without considering human factors.

#### Scalability
- Will this approach scale as volumes grow?
- Are performance implications considered?
- Is there a capacity plan?

### Accenture Additional Frameworks
- **ADM Delivery Lanes**: Different workstreams may need different methodologies (agile/waterfall/hybrid) — argument must acknowledge this complexity
- **Value Realization Tracking**: Leading indicators tracked weekly, validated impact booked monthly, Finance-validated (not self-reported)
- **Readiness Gates**: Formal criteria must be met before each phase transition — no shortcuts
- **POV Methodology**: Each recommendation pillar becomes a proof point backed by comparable client engagements (700+ references)

### What an Accenture Reviewer Would Say

**Passing argument**: "Implementable, technology-aligned, resource-realistic, and change management is considered."

**Failing argument**: "This might work in a slide deck but who's going to implement it? With what resources? How does it fit the existing architecture?"

## Cross-Lens Integration {#cross-lens-integration}

### When Lenses Conflict

Sometimes optimizing for one lens creates tension with another:

| Tension | Resolution |
|---------|-----------|
| McKinsey (structure) vs. Bain (action) | Structure serves action — restructure to make the path to action clearer |
| BCG (scenarios) vs. Deloitte (evidence) | Scenarios must be evidence-based, not speculative |
| Bain (speed) vs. Accenture (thoroughness) | Phased approach — fast first step with thorough follow-up |
| McKinsey (elegance) vs. Deloitte (completeness) | Banking context: completeness wins over elegance |
| BCG (trade-offs) vs. client expectations (perfect solution) | Acknowledge trade-offs but frame chosen option as optimal |

### Composite Quality Score Calculation

```
Overall = (McKinsey × 0.25) + (BCG × 0.20) + (Bain × 0.15) + (Deloitte × 0.25) + (Accenture × 0.15)
```

For Japanese banking context, McKinsey and Deloitte are weighted higher because:
- McKinsey: MECE and structure are paramount in Japanese business
- Deloitte: Evidence and compliance are critical in banking

### Per-Firm Quick Validation Checklist

**McKinsey**: (1) Answer stated first? (2) MECE supporting points? (3) Quantitative evidence? (4) Horizontal logic passes? (5) Vertical logic passes? (6) "So What?" at every level? (7) Hypotheses falsifiable?

**BCG**: (1) SCQA narrative arc? (2) Urgency before recommendation? (3) Novel insight (not just confirmation)? (4) Strategic options presented? (5) Multiple time horizons? (6) Environmental uncertainty considered?

**Bain**: (1) Quantified impact (dollar, timeline)? (2) Concrete implementation plan? (3) Adoption risks identified? (4) Value bridge traceable? (5) Partnership tone? (6) Key influencers identified?

**Deloitte**: (1) Recognized methodology backing? (2) Explicit acceptance criteria? (3) Risks documented with mitigations? (4) Multi-domain integration? (5) Audit-survivable? (6) Strategic Choice Cascade answered?

**Accenture**: (1) Delivery methodology per workstream? (2) Readiness gates defined? (3) Value realization tracked (leading/lagging)? (4) Scale evidence from comparable engagements? (5) Finance-validated impact? (6) Governance artifacts documented?

### Red Team Stress Test Protocol

Adapted from McKinsey's Obligation to Dissent:
1. **Assumption test**: List every assumption; challenge with "What if the opposite were true?"
2. **MECE test**: Gaps? Overlaps?
3. **"So What?" chain**: Ask "So What?" 3 times to reach actionable implication
4. **Devil's advocate**: Assign opposing position; argument must survive
5. **Implementation reality check**: "Who does this, when, and what stops them?"

## Grading Rubrics {#grading-rubrics}

### Grade A (90-100) — Exceeds Partner Standards
- Argument would be approved for client presentation without changes
- Evidence is comprehensive and verifiable
- Structure is clean, MECE, and passes all "So What?" tests
- Scenarios are robust across all cases
- Implementation path is clear and resourced
- Cultural calibration is flawless

### Grade B (75-89) — Professional Quality
- Argument is strong with minor improvement opportunities
- Evidence is solid but could have 1-2 more data points
- Structure is good but one supporting point could be clearer
- Most scenarios covered but one edge case missed
- Implementation is feasible with minor gaps
- Cultural calibration is good with minor notes

### Grade C (60-74) — Adequate But Notable Weaknesses
- Core argument works but presentation needs improvement
- Some claims lack evidence
- Structure has minor MECE issues
- Key scenario not tested
- Implementation has significant questions
- Cultural calibration has one concerning element

### Grade D (40-59) — Below Standard
- Argument has logical gaps
- Multiple claims unsupported
- Structure is unclear or non-MECE
- Only ideal scenario considered
- Implementation is unrealistic
- Cultural missteps present

### Grade F (0-39) — Fundamental Issues
- Argument is logically flawed
- No evidence base
- No discernible structure
- Would fail in any scenario
- Completely impractical
- Cultural calibration absent or offensive
